# Shopping — architecture

**Status:** active
**Last reviewed:** 2026-05-16

## Routes
| Route | File | Type |
| --- | --- | --- |
| `/shopping` | `src/app/shopping/page.tsx` | page (client component) |
| `/shopping` (loading skeleton) | `src/app/shopping/loading.tsx` | loading |
| `/shopping` (server actions) | `src/app/shopping/actions.ts` | server-actions |
| `/shopping?view=bills` | `src/app/shopping/_bills-tab.tsx` | internal tab component (mounted by `page.tsx`) |

`/shopping` is the only route under `src/app/shopping/`. The page is a `"use client"` component that drives a two-tab UI ("List" / "Bills") via the `?view=` query param; tab state is held in the URL so `router.replace` swaps tabs without scroll jump. The List tab renders the shopping-list surface (this spec); the Bills tab is implemented by `_bills-tab.tsx`, which renders bill rows owned by `features/bills.md` and embeds the bill-upload form from `features/scans.md` (`@/app/inventory/new/_bill-form.tsx`). See "Bills tab integration" below.

## Server actions

| --- | --- | --- | --- | --- |
| `addShoppingItem` | `src/app/shopping/actions.ts:28` | `{ name: string (1–120, trimmed), quantity: number (>0, required), unit?: string\|null (1–24), notes?: string\|null (≤500) }` via Zod `AddInput` | `ShoppingActionResult<{ itemId: string; alreadyExists?: boolean }>` (discriminated `ok` union; error codes `SHOPPING_INVALID`, `SHOPPING_FORBIDDEN`). | `searchShoppingItems` | `src/app/shopping/actions.ts:80` | `{ query: string (1–120, trimmed) }` via Zod `SearchInput`; on parse failure returns `{ ok: true, data: [] }` (typeahead never raises a validation error to the UI) | `ShoppingActionResult<ShoppingSearchResult[]>` where each row = `{ id, name, quantity, unit, notes, boughtAt }`. | `setShoppingItemChecked` | `src/app/shopping/actions.ts:161` | `{ itemId: uuid }` | `ShoppingActionResult<{ itemId: string }>`. | `clearShoppingItemChecked` | `src/app/shopping/actions.ts:176` | `{ itemId: uuid }` | `ShoppingActionResult<{ itemId: string }>`. Nulls `checked_at` only where `bought_at IS NULL`. | `src/components/shopping/item-row.tsx` (`onToggle`, when row is currently checked) |
| `deleteShoppingItem` | `src/app/shopping/actions.ts:191` | `{ itemId: uuid }` | `ShoppingActionResult<{ itemId: string }>`. Hard delete (no bought-state guard at the action layer — bought rows can be removed from history this way). | `src/components/shopping/edit-item-sheet.tsx` (`remove`) |
| `autoAddFromPlans` | `src/app/shopping/actions.ts:205` | _(no input)_ | `ShoppingActionResult<{ insertedCount: number; insertedNames: string[] }>`. Every action calls `requireHousehold()` (from `src/lib/auth/require.ts`) to resolve the active Clerk session → profile → household before touching Supabase. Mutating actions call `revalidatePath("/shopping")` so the route re-renders after the change (the client page's `refresh()` also re-fetches via the RLS-scoped browser client). Action results are a discriminated union (`{ ok: true; data } | { ok: false; error: { code, message, fieldErrors? } }`) — the convention shared across the codebase.

### Bills tab integration

`_bills-tab.tsx` is rendered by `ShoppingPage` when `?view=bills`. It fetches `bills` rows via the browser Supabase client (RLS-scoped to the active household), renders an `UploadBillForm` at the top for owners/maids, and maps each row to a `BillCard`. The tab exists in this route because we retired the standalone `/bills` index in `2026-05-16-bills-cleanup-and-shopping-tab-design.md`; the `bills` table, OCR ingest, and `bill_line_items` are all owned by `features/bills.md` (and the upload-form/scan pipeline by `features/scans.md`). This spec is responsible only for the tab's placement and role-gated upload affordance.

## DB surface
| Object | Kind | Introduced in | Notes |
| --- | --- | --- | --- |
| `shopping_list_items` | table | `20260526_001_shopping_list_items.sql` | Per-household standing list row. Columns: `id`, `household_id` (FK `households` ON DELETE CASCADE), `item_name` (CHECK length 1–120), `quantity` (`numeric`, CHECK `null OR > 0`), `unit` (CHECK length 1–24), `notes` (CHECK length ≤500), `bought_at`, `bought_by_profile_id` (FK `profiles` ON DELETE SET NULL), `created_by_profile_id` (FK `profiles` ON DELETE SET NULL — NOT NULL relaxed to allow the SET NULL behaviour despite the foundations pattern), `created_at`, `updated_at`. Trigger `sli_touch_updated_at` keeps `updated_at` fresh. CHECK `sli_bought_consistency` enforces `bought_by` set ⇒ `bought_at` set. Partial indexes: `sli_household_unbought_idx (household_id, created_at desc) WHERE bought_at IS NULL`, `sli_household_bought_idx (household_id, bought_at desc) WHERE bought_at IS NOT NULL`. RLS: read = active membership of `household_id`; insert/update/delete = `is_active_owner_or_maid(household_id)`. |
| `shopping_list_items.checked_at` | column | `20260630_001_shopping_checked_state.sql` | Three-state lifecycle marker. `pending` = both null; `checked` = `checked_at NOT NULL AND bought_at IS NULL` (renders on main list with strikethrough); `bought` = `bought_at NOT NULL` (moves to "Show bought" history). Partial index `sli_household_checked_idx (household_id, checked_at) WHERE checked_at IS NOT NULL AND bought_at IS NULL` keeps the daily sweep cheap. |
| `shopping_auto_add_from_plans()` | RPC (SECURITY INVOKER) | `20260527_001_shopping_auto_add_fn.sql`; rewritten by `20260622_001_ingredient_aliases.sql` (alias resolution) and `20260630_001_shopping_checked_state.sql` (no-NULL-quantity rule) | Aggregates the next 7 days of `meal_plans` rows for the caller's active household, joins to `recipe_ingredients`, resolves each `item_name` through `ingredient_aliases` (processed→shoppable), groups by `(lower(item_name), unit)`, sums quantities (treating null quantities as 1 since the 20260630 rewrite), and inserts one `shopping_list_items` row per group that does not already exist as an unbought row. Returns the inserted rows. `EXECUTE` granted to `authenticated`; uses `current_household_id_for_caller()` + `current_profile_id()` for scoping. |
| `ingredient_aliases` | table | `20260622_001_ingredient_aliases.sql` | `(processed_name PK CHECK length 1–120, shoppable_name CHECK length 1–120, created_at)`. RLS: read-only to `authenticated`. Writes happen via migrations (service-role / postgres bypass). Seeded with 7 mappings (`boiled potato → potato`, `boiled chickpeas → chickpeas`, `cooked rice → rice`, `roasted peanuts → peanuts`, `minced chicken → chicken thighs`, `minced pork → pork shoulder`, `grated coconut → coconut`); the seed `INSERT … ON CONFLICT DO UPDATE` makes re-running the migration idempotent. Resolution is applied inside `shopping_auto_add_from_plans` only — the manual `addShoppingItem` path is intentionally untouched. |
| `shopping_commit_to_inventory(p_shopping_id uuid, p_actor uuid)` | RPC (SECURITY DEFINER) | `20260630_001_shopping_checked_state.sql` | Idempotent merge: if an `inventory_items` row exists for `(household_id, lower(item_name), unit)` (via `inventory_lookup`), bumps its quantity; otherwise inserts a new row. Then stamps the shopping row's `bought_at = now()`, `bought_by_profile_id`, and `checked_at` (if null). No-ops if the row is already bought. Returns the resulting `inventory_items.id`. `EXECUTE` granted to `authenticated, postgres`. Consumed by the bill-match path in `features/bills.md` and by `shopping_sweep_checked`. |
| `shopping_sweep_checked()` | RPC (SECURITY DEFINER) | `20260630_001_shopping_checked_state.sql` | End-of-day worker. Loops every `(checked_at NOT NULL AND bought_at IS NULL)` row across all households and calls `shopping_commit_to_inventory` for each. Returns the row count. `EXECUTE` granted to `postgres` only; the daily cron route runs it via the service-role client. |
| `bill_line_items.matched_shopping_item_id` | column (FK) | `20260528_001_bills_and_line_items.sql` | Links a bill line back to the shopping row it satisfied. The bill ingest RPC in `ingest_bill_ocr` (`20260530_001_ingest_bill_ocr_fn.sql`) stamps `bought_at = bill_date` on the matched shopping row — owned/described by `features/bills.md`; cited here because it's the second path (besides the cron sweep) that moves a row from `checked` → `bought`. |
| _Bought immutability_ | _enforced in the action layer, not the DB_ | `src/app/shopping/actions.ts` | `updateShoppingItem` reads the row first and returns `SHOPPING_ITEM_BOUGHT_IMMUTABLE` if `bought_at IS NOT NULL`. `setShoppingItemChecked` / `clearShoppingItemChecked` add `.is("bought_at", null)` to the UPDATE filter so they silently no-op on bought rows. `deleteShoppingItem` does **not** guard on `bought_at` — bought rows can be removed from history via the edit sheet's Remove button. There is no DB trigger preventing direct updates to a bought row through other code paths (see Open questions). |

## External integrations
- **Clerk:** every shopping server action authenticates via `requireHousehold()` (`src/lib/auth/require.ts`), which resolves the Clerk session → `profiles` → active `household_memberships` row. The action result discriminator is then derived from RLS errors raised by Supabase against `is_active_owner_or_maid()`. The client `ShoppingPage` does not call Clerk directly — it pulls role off `household_memberships` via the browser Supabase client (which itself authenticates via the Clerk JWT).
- **Supabase:**
  - RLS-scoped server client (`createClient` from `src/lib/supabase/server.ts`) for every server action — CRUD on `shopping_list_items`, plus the `shopping_auto_add_from_plans` RPC.
  - RLS-scoped browser client (`useSupabaseClient` from `src/lib/supabase/client.ts`) for the page's read-side fetches: unbought rows, 7-day bought slice, role lookup; the `BillsTab` uses the same client to read `bills`.
  - Service-role client (`createServiceClient` from `src/lib/supabase/service.ts`) only inside the cron route (`src/app/api/cron/sweep-checked-shopping/route.ts`) to invoke `shopping_sweep_checked()`. The RPC is SECURITY DEFINER and re-asserts per-row household scoping via `shopping_commit_to_inventory`.
- **Vercel Cron:** `src/app/api/cron/sweep-checked-shopping/route.ts` runs once daily (per the design doc, 00:05 SG / 16:05 UTC), authenticated via `Bearer $CRON_SECRET`. Cron schedule itself is declared in `vercel.json` and described in `features/infrastructure.md`.

## Open questions
- **Bought immutability has no DB-level guard.** The rule lives in `updateShoppingItem` (and the partial-filter `.is("bought_at", null)` in the checked-state actions). Any other code path — a future maintenance script, a different action, a direct SQL write that satisfies RLS — can mutate a bought row's `item_name`, `quantity`, etc. If we treat bought rows as committed history (the comment in `actions.ts` explicitly does), a BEFORE UPDATE trigger that rejects column changes when `OLD.bought_at IS NOT NULL` would make the invariant load-bearing in the schema rather than implicit in one server action.
- **`deleteShoppingItem` does not guard against deleting bought rows.** The action description says "Hard delete (no bought-state guard at the action layer)" — but if bought rows are history, deletion is also a mutation. Today the edit sheet only opens for unbought rows (the page passes `onEdit` only for unbought items), so this is reachable only via a direct action call. Either tighten the action or document the intent.
- **Role-derivation in `ShoppingPage` picks the most-recently-joined active membership.** If a profile has memberships in multiple households (rare but possible), the role shown for `/shopping` may not correspond to the household whose rows RLS is actually scoping reads to. Worth aligning with whichever "current household" abstraction the rest of the app uses (e.g. `requireHousehold()` server-side).
