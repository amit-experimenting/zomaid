There is no `/bills` index page and no `/bills/new` route any more — both were removed by `2026-05-16-bills-cleanup-and-shopping-tab-design.md`. The bill list now lives in the Bills tab on `/shopping` (owned by `features/shopping.md`, file `src/app/shopping/_bills-tab.tsx`). The only canonical entry points into this feature today are:

- `/shopping?view=bills` (list) → `/bills/[id]` (detail) — owned by `features/shopping.md` upstream, owned here downstream.
- `/inventory/new?mode=scan` (upload form) → `/api/bills/scan` (OCR) → `BillConfirmForm` → `uploadBillFromScan` → `/bills/[id]` — the upload form mount and the OCR endpoint are owned by `features/scans.md`; the confirm form, the server action, and the detail page are owned here.
- `/scans/pending` (queued-retry review) → `BillConfirmForm` → `uploadBillFromScan` → `/bills/[id]` — the review page is owned by `features/scans.md`.

The `src/app/api/bills/scan/*` files live under a `bills`-prefixed path for historical reasons (the bills feature was the first OCR consumer), but the route, its parser, Sonnet client, and storage-path helper are all owned by `features/scans.md`. They are listed here as a courtesy locator because every read/write path into this feature passes through them.

Every action calls `requireHousehold()` (`src/lib/auth/require.ts`) to resolve the Clerk session → `profiles` → active `household_memberships`. `uploadBillFromScan` additionally gates on `ctx.membership.role IN ('owner','maid')`. Per-table RLS (`bills_*` / `bill_line_items_*` policies) is the second line of defence.

Revalidation: `updateBillLineItem` and `deleteBill` revalidate `/shopping` only — they intentionally do not revalidate the per-bill page (`/bills/[id]`) because they don't know the `bill_id` without an extra round-trip; the detail page re-fetches on next visit. `ingestBillLineItem` / `skipBillLineItem` / `unskipBillLineItem` revalidate `/shopping`. `uploadBillFromScan` revalidates `/shopping`, `/bills/${billId}`, `/inventory`, and (if `attemptId` provided) `/scans/pending`.

Cross-feature server-action consumers: `uploadBillFromScan` directly calls `createInventoryItem` (owned by `features/inventory.md`) as the "create new inventory row at save time" fallback for each line the user checked **Add to inventory**. The three line-item-status RPCs (`inventory_bill_ingest` / `inventory_bill_skip` / `inventory_bill_unskip`) live in `features/inventory.md`; the wrappers here are the only callers.

`uploadBillFromScan` also reaches into `shopping_list_items` directly: after the bill is persisted, it loops each clean line and marks any unbought rows matching `(household_id, ilike item_name, unit)` as `bought_at = now()`, `bought_by_profile_id = ctx.profile.id`. This is best-effort (errors are logged, not bubbled). The shopping-list table is owned by `features/shopping.md`; bills is the writer in this path.

When the caller is finalising a queued retry (i.e. `attemptId` is set), `uploadBillFromScan` opens a service-role client (`src/lib/supabase/service.ts`) for two reasons: (1) to copy the image from the `bill-scan-pending` bucket into `bill-images` so `/bills/[id]` can render it via the bill-images signed-URL path, and (2) to stamp `bill_scan_attempts.reviewed_at` + `produced_bill_id` (the table has no user-writable RLS policy by design — see `features/scans.md`).

`src/app/bills/_dedupe.ts` is a pure, unit-tested helper (no React, no Supabase). Exports `buildBillDedupeKey({ store_name, bill_date, lines })` → `DedupeKey | null` (lower-trimmed store + `YYYY-MM-DD` date + sorted `[{name, qty, unit, price}]`; returns `null` when store or date is missing so the caller skips dedupe) and `areDedupeKeysEqual(a, b)`. Used only inside `uploadBillFromScan`'s pre-insert duplicate check (§1 of that function). No fuzzy matching — exact tuple equality on the canonical key.

## External integrations
- **Clerk:** every server action and `/bills/[id]` go through `requireHousehold()` (`src/lib/auth/require.ts`). The detail page's per-row Edit/Delete affordances and the inventory-queue confirm/skip buttons gate on `ctx.membership.role` (`family_member` is read-only; owner/maid can write).
- **Supabase:**
  - RLS-scoped server client (`createClient` from `src/lib/supabase/server.ts`) for all reads on `/bills/[id]` and for every server-action write (`updateBillLineItem`, `deleteBill`, `uploadBillFromScan`'s `bills` + `bill_line_items` inserts, the shopping-match update, the inventory-queue RPC wrappers).
  - Service-role client (`createServiceClient` from `src/lib/supabase/service.ts`) used by `uploadBillFromScan` only on the queued-retry finalise path: to download from `bill-scan-pending` + re-upload to `bill-images` (which the user can do under their own RLS, but the source bucket is service-only) and to stamp `bill_scan_attempts.reviewed_at` + `produced_bill_id` (table has no user-writable RLS policy by design — see `features/scans.md`).
  - Supabase Storage signed URLs: `BillDetailPage` mints a 1-hour signed URL for `bill-images/<path>` per page render. Falls back gracefully when `image_storage_path` is the `'bill-scan-not-persisted'` sentinel or when the file is missing.
- **Anthropic (Claude Sonnet 4.6 vision):** indirect only. Bills doesn't call the model itself; the OCR happens inside `src/app/api/bills/scan/route.ts` (owned by `features/scans.md`) and the parsed payload is handed to bills through `BillConfirmForm` → `uploadBillFromScan`.

## Open questions
- **`/bills` index and `/bills/new` are fully gone — no orphans.** Verified during inventory: the only files under `src/app/bills/` are `[id]/`, `actions.ts`, and `_dedupe.ts`. The Bills list now lives at `src/app/shopping/_bills-tab.tsx` (owned by `features/shopping.md`). No half-built route to clean up.
- **`deleteBill` server action has no UI caller.** Exported, Zod-validated, RLS-guarded, but neither the detail page nor the shopping Bills tab mounts a "Delete bill" button. Either ship a destructive affordance on `/bills/[id]` (probably gated to `owner`) or remove the action.
- **`ingest_bill_ocr` RPC is dead code.** The GitHub-Issues OCR pipeline retired with `drop_bills_github_columns` (`20260628_001`) and `bills-cleanup-and-shopping-tab-design`. Nothing in app code invokes the function any more, but it still ships in production with `EXECUTE` granted to `postgres` + `service_role`. Either drop it in a follow-up migration or rewrite it as the canonical bill-ingest entry point and have `uploadBillFromScan` call it (would replace the current in-action insert sequence with one server-side transaction).
- **`bill_status` `failed` is unreachable from app code.** `uploadBillFromScan` writes `'processed'` directly; the GitHub-pipeline writer that used to land `failed` rows is gone. `BillDetailHeader` and `StatusBadge` still render the `failed` state. Either trim the enum (and the badge variants) or wire a path that can set it (e.g. when a queued retry hits 3 failures, mark any bills it was meant to produce as `failed`).
- **`status_reason` is read but never written by app code.** `BillDetailHeader` renders it as a destructive callout under the title, but `uploadBillFromScan` never sets it (and the historical writers are gone). Will stay `null` for every bill created from now on.
- **`BillDetailHeader.githubIssueUrl` is dead prop.** `BillDetailPage` always passes `null`; the column it would have read (`bills.github_issue_url`) was dropped in `20260628_001`. Remove the prop and the conditional render.
- **`updateBillLineItem` does not revalidate `/bills/[id]`.** Action comments call out the trade-off (no `bill_id` known without an extra round-trip). The detail page re-fetches on next visit, but in-place editing of a row shows stale data until the user navigates away and back. Either pass the `billId` through `LineItemEditor` so the action can revalidate the right path, or do a client-side optimistic update.
- **`uploadBillFromScan` rolls back the bill on a line-item insert failure but not on a shopping-match failure.** The shopping-match loop is explicitly best-effort; a line-item insert failure does `delete from bills where id = ...`. That's defensible (the bill is the durable record; matched shopping rows are convenience), but worth documenting against a possible future change.
