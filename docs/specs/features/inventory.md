Note: `/inventory/new` runs in three sub-modes off the same file:
- **manual** (default) — single-item form posting a server action `submitSingle` colocated in `page.tsx` that wraps `createInventoryItem`.
- **scan** (`?mode=scan` or `?mode=bill`) — renders `UploadBillForm` (`_bill-form.tsx`). That form drives the bill-scan API (`/api/bills/scan`) and on success hands off to `BillConfirmForm` (owned by `features/bills.md` / scans pipeline). Inventory only owns the file that mounts this form; the OCR pipeline + bill write-side belongs to `features/bills.md` / `features/scans.md`.
- **`?onboarding=1`** — renders `OnboardingInventoryForm` (`_onboarding-form.tsx`) which submits to `createInventoryItemsBulk`. The 15 starter rows plus user-added custom rows are parsed by `parseOnboardingFormData` and inserted one row at a time via `createInventoryItem`. After save, the action `redirect`s to `/dashboard`. This is the path linked from `InventoryPromptCard` on the dashboard.

There is no `src/app/api/inventory/route.ts`. The only inventory-shaped API folder is `src/app/api/inventory/scan/_parse.ts` — a helper module (no `route.ts`) that defines the model-response coercion used by the receipt-scan vision flow. The runtime bill-scan endpoint actually lives at `src/app/api/bills/scan/route.ts` and is owned by `features/scans.md` / `features/bills.md`; the inventory-side parse helper here predates the consolidation and is currently orphaned (see Open questions).

Every action calls `requireHousehold()` (from `src/lib/auth/require.ts`) to resolve the active Clerk session → `profiles` → active `household_memberships` row before touching Supabase. `createInventoryItem` also passes `created_by_profile_id: ctx.profile.id` so the audit field stays accurate. Write permissions are enforced at two layers: (1) the route gate in `new/page.tsx` redirects non-owner/maid callers, and (2) the `inventory_items` / RPC RLS policies require `is_active_owner_or_maid(household_id)`. `adjustInventoryItem`, `createInventoryItem`, `updateInventoryItem` all call `revalidatePath("/inventory")` (and `/inventory/${id}` where relevant); `dismissInventoryCard` revalidates `/dashboard`. `createInventoryItemsBulk` does **not** call `revalidatePath` — the trailing `redirect("/dashboard")` is what re-renders the next surface.

Cross-feature server-action consumers of inventory: `src/app/bills/actions.ts` calls `inventory_bill_ingest` / `inventory_bill_skip` / `inventory_bill_unskip` RPCs directly (not via a wrapper here) for the bill-confirm flow, and falls back to `createInventoryItem` for "create new" bill-line-item rows. Those bill-side action wrappers are owned by `features/bills.md`; the RPCs themselves are owned by this spec.

## DB surface
| Object | Kind | Introduced in | Notes |
| --- | --- | --- | --- |
| `inventory_items` | table | `20260607_001_inventory_items.sql` | One row per `(household_id, lower(item_name), unit)` — unique index `inventory_items_household_lower_name_unit_idx` enforces this so "5 kg rice" and "200 g rice" are distinct rows; deduction reconciles them at runtime via `unit_conversions`. `quantity numeric NOT NULL DEFAULT 0` with `CHECK (quantity >= 0)` — the cook-deduct path clamps to zero rather than violating this; `inventory_manual_adjust` uses `greatest(qty + delta, 0)` likewise. `low_stock_threshold` nullable numeric, rendered as a yellow "Low" badge by both the list and detail pages. `created_by_profile_id` set to `ctx.profile.id` on insert; `updated_at` maintained by trigger `inventory_items_touch_updated_at` → `touch_updated_at()`. RLS: read by `has_active_membership`; insert/update/delete by `is_active_owner_or_maid`. |
| `inventory_transactions` | table | `20260608_001_inventory_transactions.sql` | Append-only ledger. `delta numeric NOT NULL` (signed — negative for cook/manual-down, positive for bill/manual-up), `unit text NOT NULL` (the inventory row's unit at the time the write happened), `reason public.inventory_txn_reason NOT NULL`. Optional FKs to `meal_plans(id)` (cook-deduct provenance), `bill_line_items(id)` (bill-ingest provenance), and `profiles(id)` (actor; NULL for cron). Indexes: `(inventory_item_id, created_at DESC)`, partial `(meal_plan_id)` and `(bill_line_item_id)`. RLS: **read** by `has_active_membership`; **no insert/update/delete policy** — direct writes are denied to `authenticated`. Every row originates from a `security definer` writer RPC (`inventory_cook_deduct`, `inventory_bill_ingest`, `inventory_manual_adjust`). |
| `inventory_txn_reason` | enum | `20260608_001_inventory_transactions.sql` | values: `onboarding`, `manual_adjust`, `cook_deduct`, `bill_ingest`, `undo`. (`onboarding` and `undo` are reserved — no writer currently emits these reasons; the onboarding insert path uses `createInventoryItem` which writes `inventory_items` directly without a ledger row.) |
| `meal_plans.people_eating` / `cooked_at` / `deduction_status` / `deduction_warnings` | columns | `20260611_001_inventory_column_additions.sql` | Added to `meal_plans` to support the cook-deduct flow. `deduction_status meal_deduction_status NOT NULL DEFAULT 'pending'`; partial index `meal_plans_pending_deduction_idx` on `(household_id, plan_date) WHERE deduction_status = 'pending'` is the cron sweep's driver. Owned cross-feature: `meal-plan.md` describes the planner reads (`SlotWarningBadge`, `PeoplePill`); this spec describes the writes (`inventory_cook_deduct` is the only writer to all four columns). |
| `meal_deduction_status` | enum | `20260611_001_inventory_column_additions.sql` | values: `pending`, `deducted`, `skipped`, `partial`. Set by `inventory_cook_deduct`: `skipped` when no recipe on the slot; `deducted` when all ingredients found and converted; `partial` when one-or-more `deduction_warnings` were emitted. |
| `bill_line_items.inventory_ingested_at` / `inventory_ingestion_skipped` / `matched_inventory_item_id` | columns | `20260611_001_inventory_column_additions.sql` (re-asserted defensively by `20260627_001_bill_inventory_link.sql` with `IF NOT EXISTS`) | Written by `inventory_bill_ingest` / `inventory_bill_skip` / `inventory_bill_unskip`. The `matched_inventory_item_id` FK is `ON DELETE SET NULL` so a deleted inventory row doesn't cascade-delete bill history. Partial index `bill_line_items_pending_inventory_idx` on `(bill_id) WHERE inventory_ingested_at IS NULL AND inventory_ingestion_skipped = false` drives the bills-detail review queue. Columns themselves live on `bill_line_items` (owned by `features/bills.md`); the RPCs that write them are owned here. |
| `households.inventory_card_dismissed_at` | column | `20260611_001_inventory_column_additions.sql` | Nullable `timestamptz`. NULL ⇒ the dashboard still shows `InventoryPromptCard` (when the household also has <5 inventory rows). Set by `dismissInventoryCard`. |
| `unit_conversions` | table | `20260610_001_unit_conversions.sql` | `(household_id, item_name, from_unit, to_unit, multiplier numeric)`. `household_id IS NULL` = Zomaid default; `item_name IS NULL` = generic. Uniqueness via a single expression index that `coalesce`s NULLs to sentinels: `(coalesce(household_id, '00000000-…'), coalesce(lower(item_name), ''), lower(from_unit), lower(to_unit))`. RLS read = defaults visible to all authenticated, household-specific rows scoped to active members; write policies require `household_id IS NOT NULL` AND `is_active_owner_or_maid` (defaults are seed-only, no UI inserts them). |
| `unit_conversions` seed | data migration | `20260612_001_unit_conversions_seed.sql` | Ships generic cup/tbsp/tsp/l/ml/kg/lb/oz mass-and-volume rows plus per-item densities for rice (basmati / jasmine / idli / cooked / flattened), dals, flours, sugars, oils, etc. ~75 rows. These are the rows the `/inventory/conversions` page lists under "Zomaid defaults". |
| `household_roster_size(p_household uuid) → int` | RPC (helper) | `20260613_001_inventory_helpers.sql` | `language sql stable security invoker`. Returns count of `household_memberships` rows where `status = 'active'`. Default for `people_eating` when the meal-plan slot doesn't override it. |
| `inventory_lookup(p_household, p_item_name, p_unit) → inventory_items` | RPC (helper) | `20260613_001_inventory_helpers.sql` | `language sql stable security invoker`. Picks the best `inventory_items` row for an ingredient: prefers same-unit, then any matching name by largest `quantity`. Called by `inventory_cook_deduct` and cross-feature by `shopping_commit_to_inventory` (`features/shopping.md`). |
| `inventory_convert(p_household, p_item_name, p_from_unit, p_to_unit, p_qty) → numeric` | RPC (helper) | `20260613_001_inventory_helpers.sql` | `language sql stable security invoker`. Walks the four-priority lookup (household+item > global+item > household+generic > global+generic) and returns `multiplier * p_qty` or `NULL` if no conversion exists. Used inside `inventory_cook_deduct` and `inventory_bill_ingest` for unit reconciliation. Powers the cross-feature `mealplan_recipe_stock_score` ranking inside `features/meal-plan.md`. |
| `inventory_cook_deduct(p_meal_plan_id uuid) → jsonb` | RPC | `20260614_001_inventory_cook_deduct.sql` | `security definer`. Locks the `meal_plans` row (`FOR UPDATE`), permission-checks (`session_user IN ('postgres','supabase_admin')` bypass for cron, else `is_active_owner_or_maid`), is idempotent against non-`pending` rows, and `skipped` for null-recipe slots. For each `recipe_ingredients` row: scales by `people_eating / recipes.default_servings`, looks up the inventory row, converts units, clamps to available `quantity`, writes a negative `inventory_transactions` row with `reason='cook_deduct'`. Sets `deduction_status` to `deducted` (no warnings) or `partial` (warnings), `cooked_at = now()`, `deduction_warnings = warnings jsonb` (`{item_name, requested_qty, deducted_qty, unit, reason: 'not_in_stock'\|'no_conversion'\|'short'}`). Returns `{ status, warnings }` or `{ status, idempotent: true }`. Granted to `authenticated`. Today no UI calls it directly — the cron sweep is the only invoker. |
| `inventory_sweep_due_meals() → int` | RPC | `20260615_001_inventory_sweep_cron.sql` | `security definer`, `REVOKE FROM PUBLIC` + `GRANT EXECUTE TO postgres` only. Loops `meal_plans` with `deduction_status = 'pending' AND plan_date BETWEEN current_date - 2 AND current_date`, computes the lock window (next slot's `meal_time - 1 hour`, dinner uses `23:59`), and calls `inventory_cook_deduct` for any meal whose window has passed. Returns processed count. |
| `inventory-sweep` cron job | pg_cron schedule | `20260615_001_inventory_sweep_cron.sql` | `*/15 * * * *` — every 15 minutes. Idempotent registration: `unschedule` first if exists, then `cron.schedule`. Runs as `postgres` so the `session_user` bypass in `inventory_cook_deduct` and the `GRANT TO postgres` on the sweep both apply. |
| `inventory_bill_ingest(p_line_item_id, p_inventory_id, p_quantity, p_unit, p_new_item_name) → inventory_items` | RPC | `20260616_001_inventory_bill_rpcs.sql` | `security definer`. Two modes: (a) `p_inventory_id IS NULL` ⇒ insert (with `ON CONFLICT (household_id, lower(item_name), unit) DO UPDATE SET quantity = quantity + excluded.quantity` so a same-name+same-unit bill re-confirm bumps the existing row); requires `p_new_item_name`. (b) `p_inventory_id IS NOT NULL` ⇒ `FOR UPDATE` lock, convert units via `inventory_convert` (raises `INV_NO_CONVERSION` on null), bump `quantity`. Always writes a positive `inventory_transactions` row with `reason='bill_ingest'` and sets `bill_line_items.inventory_ingested_at = now()`, `matched_inventory_item_id`, `inventory_ingestion_skipped = false`. Called from `src/app/bills/actions.ts:77` (`ingestBillLineItem`). |
| `inventory_bill_skip(p_line_item_id) → void` | RPC | `20260616_001_inventory_bill_rpcs.sql` | `security definer`. Sets `inventory_ingestion_skipped = true` after permission check. Called from `src/app/bills/actions.ts:101` (`skipBillLineItem`). |
| `inventory_bill_unskip(p_line_item_id) → void` | RPC | `20260616_001_inventory_bill_rpcs.sql` | `security definer`. Reverses the skip flag. Called from `src/app/bills/actions.ts:112` (`unskipBillLineItem`). |
| `inventory_manual_adjust(p_item_id, p_delta, p_notes) → inventory_items` | RPC | `20260617_001_inventory_manual_adjust.sql` | `security definer`. `FOR UPDATE` lock, requires `is_active_owner_or_maid`, computes `new_qty = greatest(qty_before + p_delta, 0)`, writes the realized delta (`new_qty - qty_before`, may differ from `p_delta` if clamped) into a `manual_adjust` ledger row with `actor_profile_id = current_profile_id()`, `notes = p_notes`. | `mealplan_set_people_eating(p_date, p_slot, p_people) → meal_plans` | RPC | `20260618_001_meal_plan_inventory_rpcs.sql` | Cross-feature: owned by `features/meal-plan.md` for the planner-write side. The inventory side reads (not writes) `people_eating` inside `inventory_cook_deduct`. |
| `is_meal_slot_locked(p_household, p_date, p_slot) → boolean` | RPC (helper) | `20260618_001_meal_plan_inventory_rpcs.sql` | `language sql stable security invoker`. Returns `now() ≥ (plan_date + meal_time) - interval '1 hour'`. Used by the three planner write RPCs (owned by `features/meal-plan.md`). The inventory sweep computes the equivalent inline rather than calling this helper, because it needs the **next** slot's meal time, not the current one. |

## External integrations
- **Clerk:** every inventory route, server action, and the dashboard's `InventoryPromptCard` go through `requireHousehold()` (`src/lib/auth/require.ts`), which resolves the Clerk session → `profiles` → active `household_memberships` and exposes `ctx.household`, `ctx.profile`, `ctx.membership`. The role gate in `new/page.tsx` and the `canWrite` gates in `page.tsx` / `[id]/page.tsx` all read `ctx.membership.role`.
- **Supabase:**
  - RLS-scoped server client (`createClient` from `src/lib/supabase/server.ts`) for every read (`inventory_items`, `inventory_transactions`, `unit_conversions`) and every write (`createInventoryItem` / `updateInventoryItem` / `deleteInventoryItem` direct table writes; `adjustInventoryItem` via `inventory_manual_adjust` RPC; `dismissInventoryCard` direct `households` update).
  - All cook-deduct, bill-ingest, and manual-adjust **writes** to `inventory_items` + `inventory_transactions` flow through `security definer` RPCs (so the user-side `INSERT` policy on `inventory_transactions` can remain absent — the table is append-only by RPC).
  - No service-role usage from anywhere in the inventory feature itself. The bill-scan endpoint that `UploadBillForm` posts to does use the service role to write the photo into storage, but that path is owned by `features/scans.md` / `features/bills.md`.
- **pg_cron:** the `inventory-sweep` job (`*/15 * * * *`, registered in `20260615_001_inventory_sweep_cron.sql`) is the only cron job this feature owns. It runs as `postgres`, calls `inventory_sweep_due_meals` → `inventory_cook_deduct`, and is the only entry point that lands cook-deduct ledger rows today.
- **Anthropic (Claude Sonnet 4.6 vision):** indirect only. `_onboarding-parse.ts` (the inventory file) is a pure `FormData` parser and does **not** call any model. The scan/vision call happens inside `src/app/api/bills/scan/route.ts` (owned by `features/scans.md` / `features/bills.md`) via `_sonnet.ts`. `src/app/api/inventory/scan/_parse.ts` defines a model-response Zod schema and unit-coercion helpers that look like a sibling of the bill parser but no current `route.ts` imports it — see Open questions.
- **`browser-image-compression`** (npm): client-only, used by `UploadBillForm` (`maxSizeMB: 1`, `maxWidthOrHeight: 1920`, `useWebWorker: true`) before the photo is posted to `/api/bills/scan`. UX optimisation; the server side still validates type and size.

## Open questions
- **`src/app/api/inventory/scan/_parse.ts` is orphaned.** The file defines `SCAN_UNITS`, `coerceUnit`, `normalizeName`, `normalizeQuantity`, and `parseScanResponse` — exactly the shape needed for a vision-prefilled inventory scan — and a unit test (`tests/unit/inventory-scan-parse.test.ts`) covers it, but no `route.ts` sits next to it and nothing imports the module from app code. The original "scan a receipt → pre-fill inventory" spec (2026-05-16) describes a path that has since been redirected through the bills scan pipeline (`/inventory/new?mode=scan` mounts `UploadBillForm` → `/api/bills/scan` → `BillConfirmForm`). Either delete this dead module or re-wire a separate inventory-only scan route that uses it; right now the test is the only thing keeping it alive.
- **`updateInventoryItem` and `deleteInventoryItem` have no UI caller.** Both actions are exported and Zod-validated, but neither the detail page nor the list page mounts an edit/delete affordance. The detail page only renders the adjust form. If the product direction is "use manual adjust + create-new for everything", we should delete these actions; if not, we need at minimum a rename/threshold/notes edit and a delete button on `[id]/page.tsx`.
- **`inventory_transactions.reason` has unused enum values.** `onboarding` and `undo` are in the enum but no writer ever emits them. Onboarding inserts use `createInventoryItem` (direct table insert, no ledger row) — so the initial-stock movement is invisible in the audit log. Undo functionality was contemplated in the 2026-05-14 spec but never landed. Either narrow the enum or implement the two paths.
- **`createInventoryItemsBulk` does not transactionalise.** It loops `createInventoryItem` per row with no `BEGIN`/`COMMIT`, so a mid-loop failure leaves the household with a partial inventory and the user lands on `/dashboard` with no error message. Acceptable for first-time onboarding (the user can revisit `/inventory` and see what landed) but worth documenting; a single bulk RPC would also avoid the round-trip-per-row latency.
- **No write path for the conversions page.** `/inventory/conversions` renders defaults + household overrides read-only. The `unit_conversions` table has RLS allowing household-scoped owner/maid writes, but there is no form or server action to insert / update / delete overrides today. Either ship the editor or hide the "your overrides" empty-state messaging.
