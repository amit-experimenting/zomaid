# Onboarding — architecture

**Status:** active
**Last reviewed:** 2026-05-16

This spec covers the **first-run onboarding flow** that takes a signed-in Clerk user with no household and lands them on `/dashboard` with an active membership — plus the **task-setup wizard** (`/onboarding/tasks` + `/onboarding/tasks/tune`) that is the first thing the freshly-created owner sees once `maid_mode` has been picked. The foundational household data model — `households`, `household_memberships`, `invites`, `redeem_invite`, the `maid_mode` enum, and the related RLS helpers — is described from the data-model side in `infrastructure.md`; this feature owns the **create paths** that insert the first row into each, plus the **setup-gate columns** (`task_setup_completed_at`, `task_setup_drafts`, `household_task_hides`) that are read by `features/dashboard.md` and `features/tasks.md`.

The unauthenticated → authenticated handoff (`/` → `/onboarding`) is owned by `src/proxy.ts` (which lists `/onboarding(.*)` as a public-after-auth route) and `src/app/page.tsx` (which redirects authenticated callers to `/onboarding` if `getCurrentHousehold()` is null, `/dashboard` otherwise). Both are infrastructure — this feature only owns what happens **inside** `/onboarding/*`.

Authorization + gate invariants enforced in-action (in addition to per-table RLS owned by `infrastructure.md`):
- `createHouseholdAsOwner` / `createHouseholdAsMaid`: caller must have no active membership; otherwise short-circuit `/dashboard`. The page's redirect guard re-checks via `getCurrentHousehold()`, so a TOCTOU race just lands them on the dashboard with a no-op.
- `saveTaskSetupPicks` / `submitTaskSetup`: owner-only (`ctx.membership.role === 'owner'`), `ctx.household.maid_mode !== 'unset'`, and `ctx.household.task_setup_completed_at IS NULL`. The same three checks live on both wizard pages as redirect guards — the action duplicates them so direct POSTs can't bypass.
- `submitTaskSetup`'s setup-gate **CAS claim**: an `UPDATE households SET task_setup_completed_at = now() WHERE id = ? AND task_setup_completed_at IS NULL RETURNING id` is the only path that flips the latch. If zero rows match, another tab won the race and the action redirects to `/dashboard` without further writes. If the subsequent inserts (`tasks`, `household_task_hides`) throw, the catch block reverts `task_setup_completed_at` to NULL so the user can retry.
- `resetTaskSetupForEmptyState`: owner-only AND `tasks WHERE household_id = ? COUNT = 0`. The zero-task guard makes this safe to expose from the dashboard — owners with real tasks can never trip it.

Revalidation:
- `createHouseholdAsOwner` and `createHouseholdAsMaid` both `revalidatePath('/dashboard')` before redirecting.
- `submitTaskSetup` `revalidatePath('/dashboard')` before redirecting.
- `resetTaskSetupForEmptyState` `revalidatePath('/dashboard')` before redirecting.
- `saveTaskSetupPicks` does **not** revalidate (the redirect target `/onboarding/tasks/tune` is `dynamic = 'force-dynamic'`, so it re-fetches the draft on its own).

Post-submit materialisation (only in `submitTaskSetup`'s success path, after the CAS claim succeeds):
1. Insert one row per entry into `tasks` (per-household task rows, with `household_id`, `title`/`notes` cloned from the standard, frequency/interval/by-day/by-month-day/due-time from the wizard, `recurrence_starts_on = today`).
2. Upsert every still-active standard into `household_task_hides` (by `(household_id, task_id)`), so even un-picked standards never seed occurrences for this household.
3. Delete the `task_setup_drafts` row.
4. Call the `tasks_generate_occurrences(p_horizon_date)` RPC (owned by `features/tasks.md`) with a 7-day horizon, so the dashboard is non-empty on the next render.

Cross-feature mounts and links:
- `/onboarding` → `/join/code` ("I have an invite" card) — owned by `features/household.md`.
- `/onboarding/tasks` is reached from `/dashboard` via `TaskSetupPromptCard` (owned by `features/dashboard.md`) for owners who picked `maid_mode` but haven't completed Stage 2 yet. The card is gated by exactly the same `(maid_mode != 'unset' AND task_setup_completed_at IS NULL)` predicate that the wizard pages use as their guard.
- The maid-onboarding flow does **not** drop the maid into the task-setup wizard. The wizard is owner-only; the maid lands on `/dashboard` and waits for the owner to claim the invite and run setup. This is by design: tasks are owner-curated, not maid-curated.
- `src/lib/auth/require.ts:requireHousehold` is the single chokepoint that redirects "signed in but no household" → `/onboarding`; it is owned by `infrastructure.md` but every page in this feature relies on the symmetric guard (`getCurrentHousehold` → `redirect('/dashboard')` if a household exists) to keep onboarding routes invisible to already-onboarded users.

| Object | Kind | Introduced in | Notes |

| `households` | table | `20260511_001_households.sql` | **Owned by `infrastructure.md`.** This feature performs the **initial insert** in both creation actions (writes `name`, `address_line`, `postal_code`, `created_by_profile_id`). | `households.maid_mode` | column | `20260705_001_household_setup_gates.sql` | **Write surface co-owned with `features/household.md` and `features/dashboard.md`.** Enum `public.maid_mode` (`unset`, `invited`, `family_run`), `NOT NULL DEFAULT 'unset'`. **Read** by both wizard pages and both wizard actions as a gate (`!= 'unset'` required to enter Stage 1). **Indirectly written** here only via the `household_memberships_sync_maid_mode` trigger (see below) — this feature doesn't `UPDATE` the column directly, but the maid-creation action inserts a `role='maid'` membership which immediately fires the trigger. | `households.task_setup_completed_at` | column | `20260705_001_household_setup_gates.sql` | **Owned here** (this feature's `submitTaskSetup` is the only path that sets it to a timestamp, and `resetTaskSetupForEmptyState` is the only path that clears it). | `household_memberships` | table | `20260512_001_household_memberships.sql` | **Owned by `infrastructure.md`.** This feature performs the **initial insert** in both creation actions (owner: `role='owner', privilege='full', status='active'`; maid: `role='maid', privilege='full', status='active'`). The active-owner / active-maid partial unique indexes are owned by `infrastructure.md`; the has-membership pre-check in both creation actions short-circuits the path so the unique-index 23505 is never the surfaced error. |
| `invites` | table | `20260513_001_invites.sql` | **Owned by `infrastructure.md`.** This feature performs an **initial insert** in `createHouseholdAsMaid` only — minting a pending owner invite (6-digit code, 32-byte base64url token, `intended_role='owner'`, `intended_privilege='full'`) so the maid can share the join URL/code with their owner. The invite is consumed downstream by `redeemInvite` (owned by `features/household.md`). The owner-creation action does **not** mint a maid invite; the owner does that from `/dashboard` (`features/dashboard.md`) or `/household/settings` (`features/household.md`). |
| `task_setup_drafts` | table | `20260705_001_household_setup_gates.sql` | **Owned here.** One row per household, PK `(household_id)` with `ON DELETE CASCADE`. Columns: `picked_task_ids uuid[] NOT NULL DEFAULT '{}'`, `tuned_json jsonb NULL` (declared but unused in v1 — Stage 2 state lives in-memory in `TuneForm` and is only persisted on submit, not on tune-step revisit), `updated_at timestamptz NOT NULL DEFAULT now()`. RLS: read + write = `is_active_owner_or_maid(household_id)` (so a maid could read but the wizard action's own owner-only guard blocks her from getting that far). Written by `saveTaskSetupPicks` (upsert) and deleted by `submitTaskSetup` + `resetTaskSetupForEmptyState`. Read by both wizard pages. |
| `household_task_hides` | table | `20260602_001_standard_tasks.sql` | **Owned by `features/tasks.md`** (standard-tasks visibility mechanism); written here in two paths — `submitTaskSetup` (upserts every active standard as hidden, so un-picked standards don't seed occurrences) and `resetTaskSetupForEmptyState` (wipes all hides for the household). Read by `tasks_generate_occurrences` (owned by `features/tasks.md`). |
| `tasks` | table | `20260531_001_tasks_and_occurrences.sql` | **Owned by `features/tasks.md`.** This feature **reads** standard rows (`household_id IS NULL AND archived_at IS NULL`) on both wizard pages — title, recurrence, due-time, assignee — to populate the picker and the tune form. **Writes** per-household clone rows in `submitTaskSetup` (one per wizard entry, with `household_id = ctx.household.id`, `title`/`notes` from the standard, recurrence + due-time from the entry, `recurrence_starts_on = today`). Counts rows in `resetTaskSetupForEmptyState` (zero-task guard). |
| `task_occurrences` | table | `20260531_001_tasks_and_occurrences.sql` | **Owned by `features/tasks.md`.** Wiped destructively (`DELETE FROM task_occurrences`) in the `20260705_001_household_setup_gates.sql` migration so the new gated flow surfaces for every existing household on next visit. Re-materialised by the `tasks_generate_occurrences` call inside `submitTaskSetup`. |
| `households_sync_maid_mode_on_join()` + `household_memberships_sync_maid_mode` trigger | function + trigger | `20260705_001_household_setup_gates.sql` | **Owned here.** AFTER INSERT trigger on `household_memberships` — when a `role='maid'`, `status='active'` membership lands (whether from `createHouseholdAsMaid` here or from `redeem_invite` for a maid joining an owner-created household), the trigger flips `households.maid_mode` to `'invited'` if it isn't already. | `tasks_generate_occurrences(p_horizon_date date) → int` | RPC | `20260705_001_household_setup_gates.sql` (rewrite of the earlier `20260601_001` / `20260602_001` versions) | **Owned by `features/tasks.md`** (the materialisation semantics). Called by this feature at the tail of `submitTaskSetup` with a 7-day horizon so the dashboard is non-empty on the redirect. ## External integrations
- **Clerk:** `getCurrentProfile()` (used by the creation actions) and `getCurrentHousehold()` (used by the wizard actions + every page guard) both resolve to the signed-in Clerk user → `profiles.id`. Pages also use `getCurrentHousehold` / `requireHousehold` (from `src/lib/auth/*`) to enforce the symmetric redirects (`/onboarding/*` → `/dashboard` once a household exists; `requireHousehold` → `/onboarding` if not). Clerk middleware (`src/proxy.ts`) lists `/onboarding(.*)` in `PUBLIC_AFTER_AUTH`, so unauthenticated users are bounced to sign-in before they can see the chooser.
- **Supabase:**
  - Service-role client (`createServiceClient` from `src/lib/supabase/server.ts`) for **every** write in this feature — both creation actions (`households`, `household_memberships`, `invites`) and both wizard actions (`task_setup_drafts`, `tasks`, `household_task_hides`, `households.task_setup_completed_at`). The wizard pages also use the service-role client for their reads: standards (`tasks WHERE household_id IS NULL`), the draft, and the membership roster join to `profiles`. Each write action re-validates owner-only + household-scope in-handler because the service-role client bypasses RLS.
  - The wizard's CAS claim on `households.task_setup_completed_at` uses the service-role client's `update().is(col, null).select()` to make the latch race-safe across tabs.

## Open questions
- **`task_setup_drafts.tuned_json` is declared but unused.** The Stage 2 form (`TuneForm`) holds tunings in `useState` only — refreshing the page resets every entry to the standard's defaults. The column was clearly planned for "save Stage 2 progress on every change", but the wizard never writes it. Decide whether to (a) wire `TuneForm` to save tunings on each change (matches what the column's name implies) or (b) drop the column in a follow-up migration.
- **No `/onboarding/loading.tsx`.** Pages elsewhere (`/dashboard`, `/household/settings`) ship loading skeletons; onboarding has none. The pages are mostly static, so the gap is minor, but the wizard's `dynamic = 'force-dynamic'` page does a roundtrip on every entry and would benefit.
- **Owner-creation does not mint a pending maid invite.** Symmetric with `createHouseholdAsMaid` minting a pending owner invite, an owner-onboarded household could pre-mint a maid invite so the dashboard's "invite maid" CTA is one click instead of two. Currently the owner has to use `/dashboard` `inviteMaidFromHome` or `/household/settings` `createInvite`. This is arguably correct because owners may not have a maid yet (the household-mode card lets them pick `family_run`), but documenting the design trade-off here.
- **The maid path always names the household `"<ownerName>'s household"`.** No way for the maid to override this later from the maid-side UI (`/household/settings` lists the name but the action surface that updates `households.name` isn't exposed in v1). Surface in `features/household.md` if/when that gets fixed; flag here only for the cross-reference.
- **The has-membership short-circuit in both creation actions is a hard `redirect`, not an error.** Means a double-submit lands on `/dashboard` cleanly, but also means a user who tries to "start a second household" from a stale tab gets silently sent to their existing one. Probably fine — v1 cannot hold two memberships per `redeem_invite`'s P0007 — but worth noting.
- **`PickForm`'s `summarise()` re-implements recurrence formatting.** The Stage 1 picker shows each standard's recurrence as a sentence ("Weekly · Mon, Wed 08:00"). This is duplicated in the tasks feature (`features/tasks.md`). If recurrence ever supports new patterns (e.g. last-Friday-of-month), both spots will need to be updated.
