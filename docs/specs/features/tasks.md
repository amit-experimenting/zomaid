There is no `/tasks` index list or per-day page any more — both routes were collapsed into `/dashboard` by `2026-05-16-unify-plan-tasks-into-home-design.md`. `src/app/tasks/page.tsx` and `src/app/tasks/[date]/page.tsx` are kept only as redirect shims so old bookmarks, the service-worker's `notificationclick` deep link (which opens `/tasks`, see web-push integration below), and inbound push-notification taps still land somewhere live. The only real Next.js routes this feature owns are the create/edit forms (`/tasks/new`, `/tasks/edit/[id]`), the admin standard-tasks console (`/admin/tasks`), and the cron API endpoint (`/api/cron/dispatch-task-pushes`).

Cross-feature entry points into this feature today:
- `/dashboard` (owned by `features/dashboard.md`) mounts the merged day-view feed, calls `tasks_generate_occurrences` lazily on each render, reads `task_occurrences` joined to `tasks`+`profiles`, and renders this feature's `OccurrenceRow` + `OccurrenceActionSheet`. The "+ New task" button there links to `/tasks/new`; each occurrence row's overflow menu links to `/tasks/edit/[taskId]`.
- `/onboarding/tasks` and `/onboarding/tasks/tune` (owned by `features/onboarding.md`) write the initial picked-task set + tunings; on submit they materialise per-household task rows + drafts and flip `households.task_setup_completed_at`. Tasks is downstream — this feature reads the gate but never writes it.
- `/household/settings` (owned by `features/household.md`) mounts `NotificationToggle` from this feature so users can opt in/out of web-push on each device.

`src/app/push/actions.ts` is listed under this feature because the only consumer of `subscribePush` / `unsubscribePush` is `NotificationToggle` (owned here) and the only purpose of having those subscriptions in `push_subscriptions` is so the task-push dispatch cron can deliver task reminders. The web-push **infrastructure** itself (VAPID keys, the `web-push` library wrapper in `src/lib/push/webpush.ts`, the service worker at `public/sw.js`, and the `push_subscriptions` table's storage shape) is documented in `infrastructure.md`; this spec describes only what tasks consumes from that infrastructure. See **Boundary with infrastructure.md** at the bottom of External integrations.

Every household-scoped action (`createTask` through `unhideStandardTask`) calls `requireHousehold()` (`src/lib/auth/require.ts`) to resolve the Clerk session → `profiles` → active `household_memberships`. Per-table RLS is the second line of defence — see DB surface for the policy split between `tasks_insert` (active member, any role) and `tasks_update`/`tasks_delete` (owner/maid only). Occurrence writes (`markOccurrenceDone` / `markOccurrenceSkipped`) only land if the caller passes `task_occurrences_write` (owner/maid). The two admin actions call `requireAdmin()` and use the service-role client because the standard-task RLS path lets only `service_role` write rows where `household_id IS NULL`. The two push actions resolve the caller via `getCurrentProfile()` (no household required — push subscriptions are per-profile, not per-household).

Revalidation: `createTask` / `updateTask` / `markOccurrenceDone` / `markOccurrenceSkipped` / `hideStandardTask` / `unhideStandardTask` all revalidate `/tasks` (now a redirect, so the effect is downstream-only — `/dashboard` re-fetches on next visit since it isn't revalidated by these actions). `updateTask` additionally revalidates `/tasks/edit/[id]`. The two admin actions revalidate `/admin/tasks` and `/tasks`. The two push actions do not revalidate (subscription state is read client-side via the Service Worker, not server-rendered).

Side effects on recurrence change: `createTask`, `updateTask` (when the `recurrence` field is present in the patch), and `unhideStandardTask` all call `supabase.rpc("tasks_generate_occurrences", { p_horizon_date: today + 7 })` synchronously after the write, so newly-created or re-enabled tasks materialise immediately without waiting for the nightly cron. `updateTask` additionally deletes future pending occurrences for the task before regenerating, so the user sees the new schedule without stale rows. `hideStandardTask` deletes per-household pending future occurrences for the hidden standard. The admin-only `archiveStandardTask` uses the service client to bypass the per-task RLS (it operates on `household_id IS NULL` rows) and clears across-household pending occurrences for the affected standard.

UI-only "one-off" shape: `RecurrencePicker` exposes a fourth `mode` value `"one_off"` that the task-form (and the admin-client) translate on submit to `frequency: "daily", interval: 1, startsOn: <date>, endsOn: <same date>` so the existing generator emits exactly one occurrence on that day. `EditTaskPage` round-trips the inverse — a task with `frequency='daily' AND interval=1 AND ends_on IS NOT NULL AND ends_on = starts_on` is read back as `mode: "one_off"`. The DB schema has no `one_off` enum value; it's purely a UI affordance.

## External integrations
- **Clerk:** every household-scoped server action and the two task form pages go through `requireHousehold()`. The two admin actions and `/admin/tasks` go through `requireAdmin()`. The two push actions go through `getCurrentProfile()` (no household required — subscriptions are per-profile). The push-dispatch cron does not use Clerk; it authenticates via a static `CRON_SECRET` Bearer token compared against `process.env.CRON_SECRET` (Vercel Cron sets this header). The cron uses `createServiceClient()` to read across all households + profiles.
- **Supabase:**
  - RLS-scoped server client (`createClient` from `src/lib/supabase/server.ts`) for every household-scoped server action and for the `/tasks/edit/[id]` + `/tasks/new` page reads.
  - Service-role client (`createServiceClient` from `src/lib/supabase/service.ts` — also re-exported via `src/lib/supabase/server.ts`) for `/admin/tasks` page read, the two admin actions (`createStandardTask` writes a `household_id IS NULL` row; `archiveStandardTask` updates one), and the push-dispatch cron route (which reads `task_occurrences` + `tasks` + `household_memberships` + `push_subscriptions` across all households and updates `task_occurrences.notified_at` and `push_subscriptions.revoked_at`/`last_used_at`).
  - RPC: `tasks_generate_occurrences(date)` called synchronously after `createTask` / `updateTask` (when recurrence is in the patch) / `unhideStandardTask` to materialise the next 7 days, called by `/dashboard` on each render to materialise through `selectedYmd + 1`, and called by the onboarding submit (service-role) to materialise after `task_setup_completed_at` flips. The nightly pg_cron `tasks-generate-and-prune` calls it with `current_date + 7` and follows with `tasks_prune_old(90)`.
- **Vercel Cron + web-push (DISPATCH side — owned here):**
  - `vercel.json` schedules `GET /api/cron/dispatch-task-pushes` every 5 minutes. The route authenticates via `Authorization: Bearer ${process.env.CRON_SECRET}`, then reads up to `BATCH_LIMIT=200` pending occurrences with `due_at <= now() AND notified_at IS NULL`. For each occurrence it: (1) looks up active owner+maid memberships for the task's household; (2) reads their active `push_subscriptions`; (3) builds payload `{ title: tasks.title, body: assignedName ? "Due now — for " + assignedName : "Due now", data: { taskId, occurrenceId } }`; (4) calls `sendWebPush()` (owned by `infrastructure.md`) per subscription; (5) on per-sub success updates `last_used_at`; on per-sub 404/410 (`gone`) marks `revoked_at`; (6) stamps the occurrence's `notified_at = now()` unconditionally after processing (even on all-failures — there is no retry queue for tasks). The service-worker `push` handler in `public/sw.js` renders the notification with `tag: data?.occurrenceId ?? 'task'` so re-delivery of the same occurrence collapses; `notificationclick` focuses an existing `/tasks` tab or opens one.
- **Boundary with `infrastructure.md`:**
  - Owned by `infrastructure.md`: VAPID key generation + env vars (`NEXT_PUBLIC_VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`), the `web-push` library wrapper at `src/lib/push/webpush.ts` (`sendWebPush(sub, payload) → { ok } | { ok: false, gone, status }`), the service-worker plumbing under `public/sw.js` (generated by Serwist), the `push_subscriptions` table's storage schema + RLS, and the cron-secret + Vercel-cron driver setup.
  - Owned here: the `subscribePush` / `unsubscribePush` server actions (the **subscription-management API surface** that tasks consumes — even though the underlying table is infra-owned), the `NotificationToggle` UI that drives them, the `/api/cron/dispatch-task-pushes` route (the **dispatch-side consumer** that picks a domain payload shape for task notifications and decides who to fan out to per household-role rules), and the `notified_at` lifecycle on `task_occurrences`. The split is: infra owns "how a push gets delivered to a browser"; tasks owns "which event becomes a push, what's in the payload, who receives it, and when we stop trying".
  - The push-subscribe path on `NotificationToggle` reads `NEXT_PUBLIC_VAPID_PUBLIC_KEY` directly (the only env-var consumer in this feature) and shapes the call to `reg.pushManager.subscribe`. That's expected — the public key is, by design, public.

## Open questions
- **`unhideStandardTask` server action has no UI caller.** The `OccurrenceActionSheet`'s "Not applicable" copy reads *"You can re-enable it from `/admin/tasks` later"*, but `/admin/tasks` is the admin console for editing standards across all households, not a per-household un-hide UI. Either ship a "Hidden standard tasks" panel on `/household/settings` (or wherever) that lists rows from `household_task_hides` for the caller's household with an Unhide button, or change the sheet copy to match reality.
- **`/tasks/page.tsx` and `/tasks/[date]/page.tsx` are pure redirect shims.** They exist only to preserve old bookmarks + the service-worker `notificationclick` deep link to `/tasks`. If `sw.js` is updated to open `/dashboard` directly (and after a grace period for old service workers to be replaced), both shims can be deleted. The `revalidatePath("/tasks")` calls scattered across `actions.ts` would also be re-pointed at `/dashboard`.
- **`OccurrenceActionSheet` "Edit task" link is shown for any non-standard occurrence, but `markOccurrenceDone` / `markOccurrenceSkipped` writes are owner/maid-only (RLS-gated).** A `family_member` tapping a row gets the sheet rendered with Mark-done / Skip buttons that will silently fail (the action returns `TASK_FORBIDDEN` and the sheet closes). The dashboard's `DayView` does set `readOnly` from `taskActionsEnabled = isOwnerOrMaid`, so the row isn't tap-able for family members today, but the action sheet itself doesn't re-check role. Should bake role into the sheet props (and hide / disable the destructive buttons).
- **`hideStandardTask` has no inverse on the OccurrenceActionSheet itself**, even though the sheet is the only place a user hides one. If a household later regrets hiding (e.g. they bought a vacuum and now want the "Vacuum carpet" standard back), there's no in-app path — see the second open question.
- **`task_occurrences.notified_at` is set unconditionally after the cron processes an occurrence — even if every push delivery failed.** This is documented as deliberate (no retry loop) but means a transient outage on the web-push provider causes occurrences to be marked-notified-but-not-delivered with no record. Consider stamping `notified_at` only when `anySent === true` and adding a `notification_attempts int DEFAULT 0` column with a cap, mirroring `bill_scan_attempts`.
- **`/api/cron/dispatch-task-pushes` reads at most 200 occurrences per run (every 5 min).** With ~14 standard tasks × N households + per-household tasks, that's fine for small N but caps out around ~2.4k pushes/hour. Worth revisiting if user base grows, or moving fan-out off the request hot path (queue + worker).
- **Two parallel cron systems materialise the same data.** `pg_cron` runs `tasks_generate_occurrences(current_date + 7)` nightly at 22:00 SGT, and every page render of `/dashboard` calls the same RPC synchronously with `horizon = selectedYmd + 1`, and every `createTask`/`updateTask`/`unhideStandardTask` calls it with `today + 7`. The `ON CONFLICT DO NOTHING` insert makes this safe but expensive on cold renders. Consider dropping the lazy `/dashboard` call once the pg_cron has run a few times in production (or guarding it by "only call if we don't already have an occurrence for `selectedYmd`").
- **No tests against the recurrence math in `tasks_generate_occurrences`.** The shape constraint on `tasks` (via `tasks_recurrence_shape` CHECK) keeps input clean, but the day-by-day modulo arithmetic for daily-interval, weekly-interval-with-byweekday, and monthly-interval is hand-rolled in plpgsql and only exercised by manual smoke tests. A focused SQL test (or per-frequency parity check) would catch a leap-year / DST off-by-one early.
