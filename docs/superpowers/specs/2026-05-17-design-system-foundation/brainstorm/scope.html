Your ask is actually 4 specs in a trenchcoat

Each is meaningful on its own. Sequencing matters because some depend on others.

A · Foundation

Design system (mobile-first)

Color, typography, spacing, touch targets, accessibility tokens — applied across the existing PWA.

  • Light/dark color tokens (currently all neutral gray)
  • Type scale tuned for small screens + multilingual scripts
  • Min 48px touch targets, AA contrast everywhere
  • Reusable patterns (cards, lists, sheets)
Depends on: nothing. Unblocks B.
B · Persona UX

Maid-tailored experience

A different home, simpler nav, task-of-the-day focus — based on what a maid actually does (cook, shop, log bills, update inventory).

  • Maid home screen vs owner dashboard
  • Big-button, icon-forward navigation
  • Voice/photo-first inputs where possible
  • Tone, density, complexity tuned per role
Depends on: A (design tokens). Owner UX can be a lighter pass.
C · Infrastructure

i18n framework + locale plumbing

How translation works: message catalogs, locale picker, persistence, server-side rendering, fallback. No translated content yet — just the wiring.

  • Library choice (next-intl, lingui, react-i18next…)
  • URL/cookie strategy, locale on user vs household
  • Date/number formatting, pluralization rules
  • Build-time vs runtime catalog loading
Depends on: nothing. Unblocks D.
D · Content

Mizo + Manipuri translations

The actual translated strings. Low-resource languages — needs native speakers, glossary, review process.

  • String extraction from English UI
  • Translator workflow (who, how, review)
  • Script considerations (Latin Mizo, Meitei Mayek vs Bengali for Manipuri)
  • Domain glossary (cooking, household, money terms)
Depends on: C (infrastructure). Mostly a content/people problem, not engineering.
Suggested sequence: A → B in parallel with C → D once C ships. Each slice gets its own spec. Brainstorming all four at once would be 4–6 hours of design questions and a 50-page document nobody reads.
